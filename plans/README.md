# Animation improvement plans

Generated by an `improve-animations` audit of the whole frontend
(`apps/web/`) on commit `ccf3b91`. The app currently has almost no motion
investment — no motion library, no `@keyframes`, no `prefers-reduced-motion`
handling, and only five bare `transition-colors` utility classes with no
explicit duration/easing. These plans add a small, consistent set of motion
tokens and apply them to the highest-leverage gaps found across the app,
scoped to match this product's restrained "glass minimalism" personality
(`CLAUDE.md` section 3) — nothing bouncy, nothing decorative, durations kept
short.

The Nav active-tab indicator was flagged during the audit but deliberately
excluded from this plan set: nav switches happen tens of times a day, which
this audit's own frequency rule treats as "remove or drastically reduce," so
adding motion there was judged not worth it.

## Recommended execution order

1. **Plan 001 first, always.** It creates `apps/web/app/lib/motion.ts` and
   the `--ease-out`/`--duration-press`/`--duration-base` tokens in
   `globals.css` that every other plan imports from. None of 002-007 will
   compile as written until 001 has landed.
2. **002 through 006 are independent of each other** and can be applied in
   any order (or in parallel by separate executor agents) once 001 is done —
   they touch disjoint files: `Button.tsx`/`Card.tsx` (002),
   `CitationChip.tsx` (003), `ChatView.tsx` (004),
   `MeetingIngestUpload.tsx` (005), and `MeetingsListView.tsx` /
   `TracesListView.tsx` / `MeetingTimeline.tsx` / `TraceStageTimeline.tsx`
   (006).
3. **007 touches `TracesListView.tsx`, the same file as 006**, but a
   different, non-overlapping region (007 only changes the results
   container's opening `className`; 006 only changes the `.map()` body
   inside it). They can land in either order — each plan's Boundaries
   section explains how to recognize and work around the other plan having
   already landed. If applying both in the same session, land 006 before
   007 simply to keep the diff easier to review, but this is not a hard
   requirement.

## Notes

- Every plan that adds an entrance transition uses the same
  `@starting-style`-based recipe (`ENTER_TRANSITION_CLASSES` /
  `LIST_ENTER_CLASSES` from `apps/web/app/lib/motion.ts`), verified to
  compile against this repo's installed Tailwind v4.3.2 before these plans
  were written — see plan 001 for the verification details.
- Plan 003 (citation chip) intentionally only animates the expand, not the
  collapse — an existing test (`CitationChip.test.tsx:23`) requires the
  collapsed content to be fully absent from the DOM, which rules out an
  always-rendered height-animation approach without rewriting that test.
  This was a deliberate scope decision, not an oversight.
- None of these plans add new dependencies, change component props/APIs (all
  new classes/styles are applied at existing call sites, plus one internal
  `motion.ts` module), or modify any test file.
